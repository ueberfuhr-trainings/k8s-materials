# Musterlösung: Übung 4 – PostgreSQL-Datenbank

Hier findest du eine vollständige, mögliche Lösung zu **Übung 4** – als YAML **und** als imperative Befehle. Es gibt auch andere gültige Wege; wichtig ist, dass die geforderten Parameter erfüllt sind.

> **`oc` ↔ `kubectl`:** Alle Befehle funktionieren identisch. Auf reinem Kubernetes ersetzt du `oc` einfach durch `kubectl`.

---

## 1. Secret (Zugangsdaten)

### Variante A — deklarativ (YAML)

```yaml
# postgres-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: postgres-secret
type: Opaque
stringData:
  POSTGRES_USER: recipes
  POSTGRES_PASSWORD: recipes-secret-pw
```

```bash
oc apply -f postgres-secret.yaml
```

> `stringData` nimmt Klartext entgegen (Kubernetes kodiert selbst nach Base64). Unter `data` müsstest du die Werte selbst Base64-kodieren.

### Variante B — imperativ

```bash
oc create secret generic postgres-secret \
  --from-literal=POSTGRES_USER=recipes \
  --from-literal=POSTGRES_PASSWORD=recipes-secret-pw
```

### Passwort zufällig generieren

Das ist sogar sauberer, weil das Passwort nirgends im Klartext in einer Datei landet. Der Wert wird nur über das Secret referenziert (nie fest eingetragen), daher ist ein Zufallswert unproblematisch:

```bash
# URL-/shell-sicher: nur Buchstaben und Ziffern, 24 Zeichen
PW=$(LC_ALL=C tr -dc 'A-Za-z0-9' </dev/urandom | head -c 24)

oc create secret generic postgres-secret \
  --from-literal=POSTGRES_USER=recipes \
  --from-literal=POSTGRES_PASSWORD="$PW"
```

Alternativen für die Passwort-Erzeugung:

```bash
openssl rand -base64 24     # enthält evtl. + / = (im Secret unkritisch)
pwgen -s 24 1               # falls pwgen installiert ist
```

> **Gut zu wissen:** Base64 ist **keine** Verschlüsselung. Wer das Secret lesen darf, kann den Wert problemlos dekodieren:
> `oc get secret postgres-secret -o jsonpath='{.data.POSTGRES_PASSWORD}' | base64 -d`
> Mehr dazu: [Sind Secrets sicher?](https://ueberfuhr-trainings.github.io/k8s-materials/docs/secrets-sicherheit.html)

## 2. ConfigMap (Datenbankname)

### Deklarativ

```yaml
# postgres-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: postgres-config
data:
  POSTGRES_DB: recipes
```

```bash
oc apply -f postgres-config.yaml
```

### Imperativ

```bash
oc create configmap postgres-config --from-literal=POSTGRES_DB=recipes
```

## 3. ConfigMap (Init-SQL)

Am einfachsten aus einer Datei `init.sql` erzeugen (imperativ):

```bash
oc create configmap postgres-init-sql --from-file=init.sql=./init.sql
```

Deklarativ (Inhalt eingebettet):

```yaml
# postgres-init-sql.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: postgres-init-sql
data:
  init.sql: |
    CREATE TABLE recipes (
        id          VARCHAR(36)   PRIMARY KEY,
        name        VARCHAR(100)  NOT NULL,
        img         VARCHAR(255),
        servings    INTEGER       NOT NULL,
        lastEdited  TIMESTAMP WITH TIME ZONE,
        duration    INTEGER       NOT NULL,
        difficulty  VARCHAR(10),
        preparation VARCHAR(2000) NOT NULL
    );

    CREATE TABLE ingredients (
        id        BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
        name      VARCHAR(100) NOT NULL,
        quantity  DOUBLE PRECISION NOT NULL,
        unit      VARCHAR(20)  NOT NULL,
        recipe_id VARCHAR(36)  REFERENCES recipes(id) ON DELETE CASCADE
    );
```

```bash
oc apply -f postgres-init-sql.yaml
```

## 4. Deployment

```yaml
# postgres-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: postgres
spec:
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:17-alpine
          ports:
            - containerPort: 5432
          env:
            - name: PGDATA
              value: /var/lib/postgresql/data/pgdata
          envFrom:
            - secretRef:
                name: postgres-secret
            - configMapRef:
                name: postgres-config
          volumeMounts:
            - name: init-sql
              mountPath: /docker-entrypoint-initdb.d
            - name: data
              mountPath: /var/lib/postgresql/data
      volumes:
        - name: init-sql
          configMap:
            name: postgres-init-sql
        - name: data
          emptyDir: {}
```

```bash
oc apply -f postgres-deployment.yaml
```

**Worauf es ankommt:**

- **`envFrom`** zieht *alle* Schlüssel aus Secret bzw. ConfigMap als Umgebungsvariablen (`POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`). Alternativ einzeln per `env` + `valueFrom: secretKeyRef` (siehe Übung 5).
- **`init-sql` als Volume** landet unter `/docker-entrypoint-initdb.d`; das offizielle Image führt `*.sql` **nur beim ersten Start** aus (wenn das Datenverzeichnis leer ist).
- **`PGDATA` + eigenes `data`-Volume (`emptyDir`):** Ohne das scheitert der Start auf OpenShift — die willkürlich vergebene User-ID darf `/var/lib/postgresql/data` nicht anpassen (`Operation not permitted`). Das `emptyDir` liefert ein beschreibbares Verzeichnis; `PGDATA=/var/lib/postgresql/data/pgdata` legt die Daten in ein selbst erstelltes Unterverzeichnis. Persistenz ist das noch nicht — siehe Nachtrag und Übung 6.

## 6. Service *(erst in Übung 5)*

Der Service wird in **Übung 5** angelegt. Mögliche Lösung:

```yaml
# postgres-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: postgres
spec:
  selector:
    app: postgres
  ports:
    - port: 5432
      targetPort: 5432
```

```bash
oc apply -f postgres-service.yaml
# imperative Alternative:
oc expose deployment postgres --port=5432 --target-port=5432
```

> Der Service-Name `postgres` ist zugleich der DNS-Name, den das Backend in der JDBC-URL nutzt (`jdbc:postgresql://postgres:5432/recipes`).

---

## Nachtrag: `emptyDir` verstehen — und der Weg zur Persistenz

**Was ist `emptyDir`?** Ein `emptyDir` ist ein leeres Verzeichnis, das **auf dem Node** angelegt wird, auf dem der Pod läuft, und dort in den Container gemountet wird. Es lebt so lange wie der **Pod**: Ein Container-Neustart (Crash/Liveness) behält die Daten, aber beim Löschen oder Umziehen des Pods sind sie weg. Es ist also **kein** persistenter Speicher — im Gegensatz zu einem PersistentVolume, das unabhängig vom Pod existiert.

**Warum `PGDATA` auf ein Unterverzeichnis?** Würde PostgreSQL direkt in das Mount-Root `/var/lib/postgresql/data` schreiben, scheitert es in zwei Fällen: (1) auf **OpenShift**, weil die willkürlich vergebene User-ID die Rechte des Mount-Roots nicht ändern darf; (2) bei manchen **PVCs**, deren Mount-Root nicht leer ist (z.B. ein `lost+found`). Durch `PGDATA=/var/lib/postgresql/data/pgdata` legt PostgreSQL sein Datenverzeichnis in ein **selbst angelegtes Unterverzeichnis**, das es besitzt und dessen Rechte es setzen darf.

**Weg zur Persistenz:** In **Übung 6** wird das `emptyDir` mit einer Zeile durch einen `persistentVolumeClaim` ersetzt — `volumeMount` und `PGDATA` bleiben unverändert.
