# PostgreSQL-Datenbank in OpenShift bereitstellen

Als Entwickler möchte ich eine PostgreSQL-Datenbank in OpenShift betreiben, deren Konfiguration sauber und sicher bereitgestellt wird, damit das Backend später auf eine echte Datenbank umgestellt werden kann.

> **Hinweis:** Diese Übung beschreibt bewusst nur das **Ziel** und die nötigen **Parameter** – nicht den Weg dorthin. Überlege selbst, mit welchen Kubernetes-/OpenShift-Ressourcen du das umsetzt.

## 🎯 Lernziele

* Du kannst eine Datenbank als Workload in Kubernetes/OpenShift betreiben.
* Du kannst die Konfiguration einer Anwendung von außen bereitstellen – getrennt nach sensiblen und nicht-sensiblen Werten.
* Du kannst eine Datei in einen Container einbringen, die nicht Teil des Images ist.

## ✅ Definition of Done

* [ ] Ein PostgreSQL-Pod läuft mit den unten geforderten Parametern.
* [ ] Die Zugangsdaten stehen **nicht im Klartext** im Deployment-Manifest.
* [ ] Der Datenbankname wird von außen konfiguriert (nicht fest im Deployment).
* [ ] Das Init-SQL wird beim ersten Start ausgeführt und die Tabellen sind angelegt (in den Logs sichtbar).

## 🪜 Aufgabe

Benötigt wird ein **Pod mit einer PostgreSQL-Datenbank**, der die folgenden Parameter erfüllt:

| Parameter | Wert |
|---|---|
| Image | `postgres:17-alpine` |
| Port | `5432` |
| Umgebungsvariable `POSTGRES_USER` | frei wählbar, z.B. `recipes` |
| Umgebungsvariable `POSTGRES_PASSWORD` | frei wählbar, z.B. `recipes-secret-pw` |
| Umgebungsvariable `POSTGRES_DB` | `recipes` |
| Init-SQL | Datei `init.sql` im Verzeichnis `/docker-entrypoint-initdb.d/` |

Dabei gilt:

* Die Zugangsdaten (`POSTGRES_USER`, `POSTGRES_PASSWORD`) dürfen **nicht im Klartext** im Deployment stehen.
* Der Datenbankname (`POSTGRES_DB`) ist unkritisch, soll aber ebenfalls von außen konfigurierbar sein.

Das Init-SQL, das beim ersten Start das Schema anlegt:

```sql
CREATE TABLE recipes (
    id          VARCHAR(36)   PRIMARY KEY,
    name        VARCHAR(100)  NOT NULL,
    img         VARCHAR(255),
    servings    INTEGER       NOT NULL,
    lastEdited  TIMESTAMP WITH TIME ZONE,
    duration    INTEGER       NOT NULL,
    difficulty  VARCHAR(10),
    preparation VARCHAR(2000) NOT NULL
);

CREATE TABLE ingredients (
    id        BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name      VARCHAR(100) NOT NULL,
    quantity  DOUBLE PRECISION NOT NULL,
    unit      VARCHAR(20)  NOT NULL,
    recipe_id VARCHAR(36)  REFERENCES recipes(id) ON DELETE CASCADE
);
```

## 💡 Hinweise

* **Init-SQL in den Container bringen:** Das offizielle PostgreSQL-Image führt beim ersten Start automatisch alle `*.sql`-Dateien aus `/docker-entrypoint-initdb.d/` aus. Eine Datei, die nicht im Image liegt, lässt sich als **ConfigMap** anlegen und als **Volume** mounten. Abstraktes Muster:

  ```yaml
  # Ausschnitt aus einem Pod-/Deployment-Template
  spec:
    containers:
      - name: my-app
        image: my-image
        volumeMounts:
          - name: my-files
            mountPath: /pfad/im/container
    volumes:
      - name: my-files
        configMap:
          name: meine-configmap
  ```

* **Kein Persistent Volume nötig:** Für diese Übung genügt flüchtiger Speicher. Produktionsreif wäre ein **StatefulSet** mit PersistentVolumeClaim – siehe [PostgreSQL auf Kubernetes – Best Practices](../../docs/postgres-as-statefulset.html).

## 🔎 Prüfung

```bash
oc get pods
oc logs <postgres-pod-name>
```

In den Logs sollte die Ausführung des Init-Skripts (`init.sql`) sichtbar sein.
